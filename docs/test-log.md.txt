# 🧪 Test Log – Century VGreen Pool Pump Controller

This document tracks testing, failures, debugging steps, and key discoveries during development of the Arduino UNO R4 WiFi pool pump controller.

---

# 📅 Early Development – Initial Bring-Up (RS-485 & Protocol)

## 🎯 Objective

* Establish communication with pump over RS-485
* Validate EPC protocol behavior
* Confirm command/response structure

---

## 🔍 Initial Issues

### ❌ No Communication

* Pump did not respond to commands
* Occasional garbage data observed

### 🧠 Root Cause

* TX/RX lines were reversed between Arduino and RS-485 module

### 🔧 Fix

* Swapped TX/RX wiring
* Communication established successfully

---

## 🔍 Protocol Validation

### ✅ Observations

* Pump uses **custom EPC protocol (NOT Modbus)**
* CRC confirmed as:

  * **CRC-16 (0xA001 polynomial)**

---

### 🧪 Verified Commands

**STOP Command**

```id="4l5k5f"
TX: 15 42 20 51 7C
RX: 15 42 10 51 68
```

**SET RPM Command**

```id="7ndm7m"
TX: 15 44 20 00 40 1F 89 19
```

---

### 🧠 Findings

* Frame structure:

  * Address + Function + Data + CRC
* Responses include:

  * ACK / NACK behavior

---

## 🔁 Communication Behavior Discovery

### 🔍 Observations

* Pump stops running when communication stops
* Sending a **GO / RUN command alone does NOT start the pump**

---

### 🧠 Findings

* Pump requires **continuous valid communication**
* Timeout occurs at approximately:

  * **~30 seconds**
* A **valid demand (RPM setpoint)** must be established **before** the pump will run
* GO/Run command acts as:

  * Enable / maintain run state
  * NOT a start trigger by itself

---

### 🔧 Required Start Sequence (Discovered Behavior)

1. Send **SET RPM (demand)**
2. Then send **GO / RUN command**
3. Continue sending keep-alive commands

---

### 🔧 Action

* Implemented:

  * Keep-alive command loop
  * Demand-first control logic

---

# 📅 Early Control Testing (Pre-Web UI)

## 🎯 Objective

* Control pump manually via serial commands

---

## 🔍 Observations

### ⚠️ Command Sensitivity

* Some commands require:

  * Repetition
  * Timing delays

### ⚠️ Startup Behavior

* Pump does not instantly run at commanded RPM

---

## 🧠 Findings

* Pump performs internal startup sequence (**Prime Mode**)
* Prime must complete before stable operation

---

## 🔧 Design Decision

* All pump starts must:

  * Enter **Prime Mode**
  * Then transition to target RPM

---

# 📅 UI Interaction & Command Handling Issues

## 🔍 Observations

### ❌ Rapid Commands Cause Lockup

* Sending multiple commands quickly results in:

  * Controller freeze
  * Loss of communication

### ❌ UI Refresh Impact

* Page reload temporarily interrupts communication
* Can cause missed or delayed commands

---

## 🧠 Findings

* System cannot handle:

  * Back-to-back command execution
* Blocking code sections delay critical tasks

---

## 🔧 Actions

* Introduced command pacing
* Transitioned toward non-blocking logic
* Reduced UI refresh aggressiveness

---

# 📅 Ramp & Power Behavior Testing

## 🎯 Objective

* Prevent faults during RPM transitions

---

## 🔍 Observations

* Large RPM step changes cause:

  * **DC low-voltage faults**
* Ramp down more problematic than ramp up

---

## 🧠 Findings

* Power instability during rapid deceleration
* Behavior likely dependent on load conditions

---

## 🔧 Working Parameters

* Step size: **25 RPM**
* Delay: **250 ms**

⚠️ Testing performed without hydraulic load

---

# 📅 Aux Relay Control Testing

## 🎯 Objective

* Control auxiliary relay via EPC commands

---

## 🔍 Observations

* Relay behavior inconsistent
* Relay sometimes activates:

  * ~3 minutes after prime ends
* ON/OFF behavior occasionally appears inverted
* Commands often require repetition

---

## 🧠 Findings

* Relay control influenced by:

  * Stored pump configuration
  * RPM thresholds
  * Internal timing logic
* Direct commands do not guarantee immediate control

---

## 🔧 Commands Tested

* Write to:

  * Page `0x81`, Address `0x07` (Aux relay control)
* Followed by:

  * STORE CONFIG command

---

## ⚠️ Issues

* Some writes return:

  * `NACK: Data out of range`
* Writes more reliable when:

  * Pump is **stopped**

---

# 📅 UI Lockup Root Cause Identified

## 🔍 Observations

* UI loading causes:

  * Pump communication pause
  * System instability or freeze

---

## 🧠 Findings

* HTTP handling blocks main loop execution
* Serial communication starved during UI processing

---

## 🔧 Actions

* Reduced UI refresh frequency
* Minimized dynamic content
* Avoided polling-heavy design

---

# 📅 Runtime Stability Issue (~28 Minute Failure)

## 🔍 Observations

* System locks up after ~28 minutes:

  * UI inaccessible
  * Pump communication stops

---

## 🧠 Findings

* Likely causes:

  * Memory exhaustion
  * Logging overhead
  * Resource fragmentation

---

## 🔧 Actions

* Removed all logging features
* Simplified runtime loop
